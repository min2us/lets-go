Chapter 8.

Processing forms 폼 처리

In this section of the book we’re going to focus on adding an HTML form for creating new snippets. The form will look a bit like this:

이번 섹션에서는 새로운 스니펫을 생성하기 위한 HTML 폼을 추가하는 것에 중점을 둘 것입니다. 폼은 다음과 같은 형태가 될 것입니다.

The high-level flow for processing this form will follow a standard Post-Redirect-Get pattern and will work like so:

이 폼을 처리하는 고수준 흐름은 표준 Post-Redirect-Get 패턴을 따를 것이며 다음과 같이 동작할 것입니다.

  1. The user is shown the blank form when they make a GET request to /snippet/create.

    사용자가 /snippet/create 로 GET 요청을 하면 빈 폼이 표시됩니다.

  2. The user completes the form and it’s submitted to the server via a POST request to /snippet/create.

    사용자가 폼을 작성하고 /snippet/create 로 POST 요청을 통해 서버에 제출합니다.

  3. The form data will be validated by our snippetCreatePost handler. If there are any validation failures the form will be re-displayed with the appropriate form fields highlighted. If it passes our validation checks, the data for the new snippet will be added to the database and then we’ll redirect the user to "/snippet/view/:id".

    폼 데이터는 snippetCreatePost 핸들러에 의해 검증됩니다. 검증 실패 시, 적절한 폼 필드가 강조된 상태로 폼이 다시 표시됩니다. 검증을 통과하면 새로운 스니펫의 데이터가 데이터베이스에 추가되고, 사용자는 "/snippet/view/:id" 로 리디렉션됩니다.

As part of this you’ll learn:

이 과정의 일환으로 여러분은 다음을 배우게 될 것입니다.

  • How to parse and access form data sent in a POST request.

    POST 요청으로 전송된 폼 데이터를 파싱하고 접근하는 방법.

  • Some techniques for performing common validation checks on the form data.

    폼 데이터에 대한 일반적인 검증 체크를 수행하는 몇 가지 기법.

  • A user-friendly pattern for alerting the user to validation failures and re-populating form fields with previously submitted data.

    사용자에게 검증 실패를 알리고, 이전에 제출한 데이터로 폼 필드를 다시 채우는 사용자 친화적인 패턴.

  • How to keep your handlers clean by using helpers for form processing and validation.

    폼 처리 및 검증을 위한 헬퍼를 사용하여 핸들러를 깔끔하게 유지하는 방법.
