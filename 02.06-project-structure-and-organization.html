Chapter 2.6.

Project structure and organization

Before we add any more code to our main.go file, it’s a good time to think how to organize and structure this project.

main.go 파일에 더 많은 코드를 추가하기 전에, 이 프로젝트를 어떻게 구성하고 구조화할지 생각해볼 좋은 시간입니다.

It’s important to explain upfront that there’s no single right — or even recommended — way to structure web applications in Go. And that’s both good and bad. It means that you have freedom and flexibility over how you organize your code, but it’s also easy to get stuck down a rabbit-hole of uncertainty when trying to decide what the best structure should be.

가장 적합한 또는 권장되는 방법으로 Go 언어로 웹 애플리케이션을 구조화하는 유일한 올바른 방법이 없다는 것을 미리 설명하는 것이 중요합니다. 이것은 코드를 어떻게 구성할지에 대한 자유와 유연성을 의미하지만, 동시에 어떤 구조가 최선인지 결정하려고 할 때 불확실성의 미로에 빠지기 쉽다는 것을 의미합니다.

As you gain experience with Go, you’ll get a feel for which patterns work well for you in different situations. But as a starting point, the best advice I can give you is don’t over-complicate things. Try hard to add structure and complexity only when it’s demonstrably needed.

Go 언어에 대한 경험이 쌓이면 각 상황에서 어떤 패턴이 잘 동작하는지에 대한 감각이 생길 것입니다. 하지만 시작할 때 가장 좋은 조언은 과도하게 복잡하게 만들지 말라는 것입니다. 구조와 복잡성을 추가할 때 실제로 필요한 경우에만 노력하십시오.

For this project we’ll implement an outline structure which follows a popular and tried-and-tested approach. It’s a solid starting point, and you should be able to reuse the general structure in a wide variety of projects.

이 프로젝트에서는 널리 사용되고 검증된 방식을 따르는 개요 구조를 구현할 것입니다. 이는 견고한 시작점이며, 이 일반적인 구조를 다양한 프로젝트에서 재사용할 수 있어야 합니다.

이제 프로젝트 저장소의 구조는 다음과 같이 보여야 합니다.

Let’s take a moment to discuss what each of these directories will be used for.

이제 각 디렉토리가 어떤 용도로 사용될지에 대해 잠시 이야기해 보겠습니다.

  • The cmd directory will contain the application-specific code for the executable applications in the project. For now our project will have just one executable application — the web application — which will live under the cmd/web directory. "cmd 디렉토리에는 프로젝트의 실행 가능한 애플리케이션에 대한 애플리케이션 특정 코드가 포함됩니다. 현재 프로젝트에서는 cmd/web 디렉토리에 위치한 웹 애플리케이션만을 가지게 될 것입니다."

  • The internal directory will contain the ancillary non-application-specific code used in the project. We’ll use it to hold potentially reusable code like validation helpers and the SQL database models for the project."internal 디렉토리에는 프로젝트에서 사용되는 부수적이고 애플리케이션과 관련 없는 코드가 포함됩니다. 이 디렉토리는 프로젝트에서 재사용 가능한 코드를 보관하는 데 사용되며, 검증 도우미 및 프로젝트의 SQL 데이터베이스 모델과 같은 코드를 포함할 것입니다."

  • The ui directory will contain the user-interface assets used by the web application. Specifically, the ui/html directory will contain HTML templates, and the ui/static directory will contain static files (like CSS and images)."ui 디렉토리에는 웹 애플리케이션에서 사용되는 사용자 인터페이스 자산이 포함됩니다. 구체적으로 ui/html디렉토리에는 HTML 템플릿이, 그리고 ui/static 디렉토리에는 정적 파일(예: CSS 및 이미지)이 포함될 것입니다."

So why are we using this structure?"그래서 우리는 왜 이 구조를 사용할까요?"

There are two big benefits:"이 구조를 사용하는 두 가지 큰 이점이 있습니다."

  1. It gives a clean separation between Go and non-Go assets. All the Go code we write will live exclusively under the cmd and internal directories, leaving the project root free to hold non-Go assets like UI files, makefiles and module definitions (including our go.mod file). "Go 코드와 Go 이외의 자산 사이에 깔끔한 분리를 제공합니다. 작성한 모든 Go 코드는 cmd 및 internal 디렉토리에 독점적으로 존재하게 되어 프로젝트 루트는 UI 파일, makefiles 및 모듈 정의(우리의 go.mod 파일 포함)와 같은 Go 이외의 자산을 보유할 수 있게 됩니다."

  2. It scales really nicely if you want to add another executable application to your project. For example, you might want to add a CLI (Command Line Interface) to automate some administrative tasks in the future. With this structure, you could create this CLI application under cmd/cli and it will be able to import and reuse all the code you’ve written under the internal directory."이 구조는 프로젝트에 또 다른 실행 가능한 애플리케이션을 추가하려는 경우에 매우 효율적으로 확장됩니다. 예를 들어 나중에 어떤 관리 작업을 자동화하기 위해 CLI(Command Line Interface)를 추가하고 싶을 수 있습니다. 이 구조를 사용하면 cmd/cli 디렉토리 아래에 이 CLI 애플리케이션을 생성할 수 있으며, internal 디렉토리 아래에 작성한 모든 코드를 가져와 재사용할 수 있게 됩니다."

Additional information

The internal directory

It’s important to point out that the directory name internal carries a special meaning and behavior in Go: any packages which live under this directory can only be imported by code inside the parent of the internal directory. In our case, this means that any packages which live in internal can only be imported by code inside our snippetbox project directory. "Go 언어에서 internal 디렉토리 이름은 특별한 의미와 동작을 가지고 있습니다. 이 디렉토리 아래에 있는 모든 패키지는 internal 디렉토리의 부모 디렉토리 내의 코드에서만 가져올 수 있습니다. 우리의 경우에는 internal 디렉토리에 있는 패키지는 오직 snippetbox 프로젝트 디렉토리 내의 코드에서만 가져올 수 있습니다."

Or, looking at it the other way, this means that any packages under internal cannot be imported by code outside of our project."또는 다른 관점에서 보면, internal 하위의 패키지는 프로젝트 외부의 코드에서 가져올 수 없다는 것을 의미합니다."

This is useful because it prevents other codebases from importing and relying on the (potentially unversioned and unsupported) packages in our internal directory — even if the project code is publicly available somewhere like GitHub."이는 다른 코드베이스가 우리의 internal 디렉토리의 (아마 버전 관리되지 않고 지원되지 않을 수 있는) 패키지를 가져와 의존하지 못하도록 하는데 유용합니다. 심지어 프로젝트 코드가 GitHub과 같은 곳에서 공개적으로 사용 가능하더라도 말이죠."
